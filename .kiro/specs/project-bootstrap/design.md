# Design Document: Project Bootstrap

## Overview

This design translates the requirements into architectural decisions and file-level specifications for the project bootstrap. It describes *what* each file does and *why*, leaving exact file contents to the implementation tasks.

The bootstrap produces a buildable, lintable, type-safe Next.js App Router project with the directory skeleton from `steering/structure.md`. No application logic, no content processing, no UI beyond a placeholder page.

### Key Design Decisions

1. **Framework and tooling versions follow `steering/tech.md`** — The scaffolding uses the latest stable Next.js with App Router and the Tailwind version bundled by the scaffolder. Exact versions are pinned at scaffolding time, not prescribed here.
2. **Tailwind CSS-based configuration** — Current Tailwind uses CSS-native config (`@import "tailwindcss"`) rather than a JS config file. Content path scanning uses a `@source` directive in the global stylesheet to include `content/` (which lives outside `src/`).
3. **ESLint with framework-recommended config** — The scaffolder generates the ESLint config format appropriate for the Next.js version. We use `eslint-config-next` which bundles React, JSX-a11y, and Next.js-specific rules. The exact config format (flat vs legacy) is determined by the scaffolder.
4. **Prettier as standalone formatter** — Separate from ESLint, no integration plugins. Avoids rule conflicts. Two scripts: `format` (write) and `format:check` (verify without writing, CI-friendly).
5. **`@/` alias via tsconfig paths** — Configured in `tsconfig.json` under `compilerOptions.paths` as specified in `steering/structure.md`. Next.js reads it automatically.
6. **`.gitkeep` for empty directories** — One per empty skeleton directory to preserve the structure in git.
7. **Minimal root layout** — No nav, no footer, no providers. Just `<html>`, `<body>`, metadata, and global styles import. Future specs layer on top.

## Components and Interfaces

### Scaffolding Approach

The project is initialized via `pnpm create next-app` with flags for TypeScript, Tailwind, ESLint, App Router, `src/` directory, `@/*` import alias, and pnpm. This generates the base project which is then customized.

### Files from Scaffolding (with customization notes)

| File | Role | Post-Scaffolding Work |
|---|---|---|
| `package.json` | Dependencies, scripts | Add Prettier dep, add `format` and `format:check` scripts |
| `tsconfig.json` | TypeScript config | Verify `strict: true` and `@/*` paths are present |
| `next.config.ts` | Next.js config | None — minimal, no custom config for bootstrap |
| `src/app/layout.tsx` | Root layout | Simplify to minimal shell, set metadata title |
| `src/app/page.tsx` | Home page | Replace with placeholder heading |
| `src/styles/globals.css` | Global styles | Add `@source` directive for `content/` scanning |
| ESLint config | Linting rules | Verify framework-recommended ruleset is configured |
| `postcss.config.mjs` | PostCSS | None — generated by scaffolding |

### Files Created Manually

| File | Role |
|---|---|
| `.prettierrc` | Prettier configuration (repository-level) |
| `.gitkeep` files | One per empty skeleton directory |

### Design Specifications

#### Global Stylesheet (`src/styles/globals.css`)

Imports Tailwind and adds a `@source` directive pointing to `content/` (relative path from `src/styles/` to project root's `content/`). This ensures Tailwind scans content files for utility classes, satisfying R2.2.

#### Prettier Configuration

Repository-level `.prettierrc` with project conventions (semicolons, quotes, tab width, trailing commas, print width). Two package.json scripts:
- `format` — writes formatted output (developer use)
- `format:check` — exits non-zero if files don't match (CI use)

Both scripts cover `src/` and `content/` file patterns.

#### Root Layout (`src/app/layout.tsx`)

- Server Component (no `"use client"`)
- Renders `<html lang="en">` and `<body>`
- Exports metadata with title "Lorenzo Santucci"
- Imports global stylesheet for Tailwind
- No navigation, footer, or providers

#### Placeholder Page (`src/app/page.tsx`)

- Server Component
- Renders `<h1>Lorenzo Santucci</h1>` inside `<main>`
- No styling beyond Tailwind defaults

#### Path Alias

`tsconfig.json` maps `@/*` to `./src/*`. Next.js reads this automatically — no additional config needed.

### Directory Skeleton

Every directory from `steering/structure.md` that doesn't already contain a source file gets a `.gitkeep`:

- `src/components/`, `src/lib/`, `src/world/`
- `content/pages/`, `content/projects/`, `content/blog/`, `content/agents/`, `content/locations/`, `content/characters/`, `content/dialogues/`, `content/services/`
- `public/assets/`

Directories with source files (`src/app/`, `src/styles/`) don't need `.gitkeep`.

## Data Models

None. This spec produces configuration only. Content schemas and state models come in later specs.

## Correctness Properties

### Property 1: Directory skeleton completeness

*For any* directory listed in the "Directory Layout" section of `steering/structure.md`, that directory SHALL exist on disk, and if it contains no source files, it SHALL contain a `.gitkeep` file.

**Validates:** R5.1, R5.2

This is the only universally quantified property in this spec. All other acceptance criteria are single-point config checks verified through the build and verification gates.

## Error Handling

No application-level error handling. Build toolchain errors (missing deps, type errors, lint violations, missing alias targets, unscanned content paths) are caught by the verification gates and produce clear error messages pointing to the issue.

## Testing Strategy

No test runner is installed in this spec. Verification relies on build-time checks and structural assertions.

### Verification Gates

```bash
pnpm install            # Dependencies install without errors
pnpm build              # Compiles (validates strict mode, alias, Tailwind, Server Components)
pnpm lint               # Zero lint errors
pnpm format:check       # Code matches formatter config
```

A successful `pnpm build` implicitly validates TypeScript strict mode, path alias resolution, Tailwind configuration, and Server Component correctness.

### Structural Verification

Property 1 (skeleton completeness) is verified by checking that each expected directory exists and empty ones contain `.gitkeep`. When a test runner is introduced in a later spec, this should become a proper test iterating over the directory list from `steering/structure.md`.

### Config Spot-Checks

Single-point verifications for each requirement: `strict: true` in tsconfig, lockfile exists, Prettier config exists, scripts exist, `context/` and `CLAUDE.md` untouched, layout has `lang` attribute and metadata title, page has heading. These are implementation-time checks, not formal tests.
